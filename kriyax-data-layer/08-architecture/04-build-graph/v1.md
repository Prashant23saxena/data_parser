## Overview

The original foundation is already built: workspace bootstrap, DuckDB health, file import, catalog preview, Python execution, saved scripts, Odoo full import, pipeline creation, enable/disable, manual run, and run detail.

The dependency graph below therefore focuses on the remaining build. Locked L8.3 contracts convert frontend/backend dependencies into mockable soft dependencies, so several streams can now move in parallel after the existing foundation.

Recommended streams:

1. Stream A: Existing foundation and contract baseline
2. Stream B: Pipeline scheduling and failure surfacing
3. Stream C: Odoo incremental sync and pipeline pre-step
4. Stream D: Storage management and export
5. Stream E: Agentic code generation and correction
6. Stream F: Frontend polish, summaries, and integration wiring
7. Stream G: Integration and UAT hardening

Critical path for remaining Phase 3: Stream B scheduler core -> Stream C Odoo pre-step -> Stream F pipeline UI/Home failure summary -> Stream G E2E/UAT.

## Critical Path

Remaining build critical path:

1. Pipeline schedule model and validation.
2. Scheduler restore on backend startup.
3. Odoo incremental sync cursor and sync operation.
4. Pipeline connector pre-step orchestration.
5. Frontend schedule/pre-step/failure UI wiring.
6. Integration tests for scheduled pipeline with connector pre-step and failure detail.

Why this is the critical path:
- Scheduling depends on a durable PipelineSchedule model.
- Odoo pre-step depends on an OdooSyncCursor model.
- Full pipeline automation depends on both.
- Home/pipeline failure UX depends on run history/failure state.

Agent implementation is important but not on the remaining pipeline automation critical path. It can proceed in parallel once C-008 is frozen.

## Bottleneck Components

| Component | Why it bottlenecks | Mitigation |
|---|---|---|
| backend-pipelines-service | owns schedule, run orchestration, failure state, and pre-step integration | keep orchestration thin; delegate execution and Odoo sync |
| backend-catalog-service | multiple streams need metadata and table updates | preserve stable C-003; add helper functions instead of ad hoc JSON writes |
| frontend `App.tsx` | currently contains many screens in one file | allow functional completion first; split by feature module when work packages are defined |
| C-008 Agent API | controls privacy-sensitive LLM behavior | sample rows default false; user insert/run approval required |

## Risks

| # | Risk | Mitigation |
|---|---|---|
| R1 | Scheduler state and JSON schedule metadata drift | Treat JSON pipeline metadata as source of truth; restore scheduler from JSON on startup |
| R2 | Odoo incremental sync corrupts existing tables | Use cursor metadata and explicit upsert logic; test insert, update, no-change paths |
| R3 | Frontend builds ahead of backend and misses runtime edge cases | Use frozen contract mocks plus Playwright E2E against real backend before marking slices done |
| R4 | Agent sends sensitive row samples to LLM | C-008 defaults `includeSampleRows` to false; schema-only context by default |
| R5 | Large `App.tsx` slows changes | Package frontend work by screen/feature and split only when it reduces active implementation risk |
