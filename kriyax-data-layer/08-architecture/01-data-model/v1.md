# Data Model — KriyaX Data Layer

**Version:** v1
**Status:** FROZEN
**Last updated:** 2026-05-03
**Tech stack ref:** L7 build plan v1
**Implementation baseline checked:** `kriyax-data-app` Phase 1, Phase 2, and initial Phase 3 build slices

## Overview

KriyaX Data Layer uses a project-scoped local workspace as the system boundary. The current implementation already proves the right split:

- DuckDB is the source of truth for actual ingested and derived data tables.
- Workspace metadata JSON files are the source of truth for app control-plane records: catalog entries, saved Odoo connections, and pipeline definitions.
- Python scripts are stored as real `.py` files.
- Pipeline and execution run logs are append-only JSON files under workspace runs.
- Uploaded source files are retained under workspace uploads as import evidence and optional reinspection source.

This v1 model keeps that bare-bones local pattern. It does not introduce users, teams, RBAC, server-side governance, or a heavy orchestration database because L1 explicitly defines this as an internal bridge tool.

## Entities

### Workspace

**Identity:** absolute workspace root path.
**Source of truth:** local filesystem.
**Lifecycle:** created or ensured on backend startup; deleted only by operator outside the app.
**Current implementation:** `kriyax-data-app/backend/app/services/workspace.py`.

| Field | Type | Required | Default | Constraints | Notes |
|---|---|---|---|---|---|
| root_path | path | yes | app-local `workspace/` | must be writable | May be overridden by `KRIYAX_WORKSPACE_ROOT`. |
| uploads_path | path | yes | `data/uploads` | must be writable | Stores uploaded CSV/XLSX files. |
| exports_path | path | yes | `data/exports` | must be writable | Required by export feature. |
| scripts_path | path | yes | `scripts` | must be writable | Stores saved Python scripts. |
| runs_path | path | yes | `runs` | must be writable | Stores execution and pipeline run records. |
| warehouse_path | path | yes | `warehouse` | must be writable | Contains DuckDB file. |
| metadata_path | path | yes | `metadata` | must be writable | Contains JSON control-plane files. |

**Relationships:**
- has_one Warehouse.
- has_many UploadedFiles, SavedScripts, ScriptExecutions, PipelineRuns.
- has_many metadata sidecars: CatalogEntry, OdooConnection, Pipeline.

**Access patterns:**
- Ensure workspace at app start.
- Resolve paths for every backend service.

### DataTable

**Identity:** `schema.table_name` within DuckDB.
**Source of truth:** DuckDB.
**Lifecycle:** created by file import, Odoo import, or `save_table()`; renamed/truncated/dropped by table management; replaced by derived-table saves.

| Field | Type | Required | Default | Constraints | Notes |
|---|---|---|---|---|---|
| qualified_name | string | yes | `schema.table` | unique in warehouse | Primary external identifier. |
| schema | string | yes | user-selected or helper default | DB-safe | Examples: `raw_files`, `odoo`, `curated`. |
| table_name | string | yes | user-selected/helper | DB-safe | |
| row_count | integer | yes | computed | >= 0 | Stored in catalog snapshot, live count may drift. |
| column_count | integer | yes | computed | >= 0 | |
| created_at | datetime | yes | now | UTC ISO timestamp | Stored in catalog entry. |

**Relationships:**
- has_one CatalogEntry.
- has_many CatalogColumns.
- may be read by SavedScripts and AgentGeneratedCode.
- may be created by UploadedFile, OdooConnection/OdooModel, or SavedScript.

**Access patterns:**
- List catalog tables.
- Preview table rows.
- Load table into pandas with `load_table(name)`.
- Save pandas DataFrame into DuckDB with `save_table(df, name, schema)`.
- Export as CSV.

## Cross-Cutting Decisions

### Auth and tenancy

No user, organization, session, RBAC, or tenant entities in v1. L1 says the app is internal to the Kriya team and not client-facing. The workspace itself is the operational boundary.

### Credentials

Odoo credentials are project-local metadata in `connections.json` for v1. API responses must mask secrets. If the workspace leaves a trusted laptop/server, add encrypted secret storage before multi-user deployment.

### Audit/history

No general audit log in v1. PipelineRun and ScriptExecution outputs are the operational history because the app's highest-risk workflows are scheduled jobs and script execution. Table management actions should add lightweight event records when F-DS-004 is implemented.

### Soft delete

No soft delete for DataTable in v1. Dropping a table is destructive and must require confirmation. Pipelines and saved connections should prefer disabled/unused states where possible; deletion warnings are required when referenced by pipelines.

### Schema evolution

Connector imports are new-table-only in v1. Derived table saves may replace an existing derived table and should upsert the catalog entry rather than creating duplicates. Rich schema version history is parked to backlog.

### Privacy and PII

Client data may include PII inside DataTables, but the app is a local internal bridge tool. Do not send sample rows to the agent unless explicitly approved in the agent implementation step. Schema metadata is safe default context; row samples are opt-in.

### Scheduler restart

PipelineSchedule must be persisted in pipeline JSON and restored into the scheduler on backend startup. Runtime scheduler state is not the source of truth.

## Implementation Alignment Notes

The current build already matches the model in these areas:

- Workspace bootstrap and DuckDB health.
- File upload inspection and editable column mapping.
- File import into DuckDB plus catalog registration.
- Catalog table listing and preview.
- Saved Python scripts as `.py` files.
- `load_table()` and `save_table()` helpers.
- Derived table catalog registration.
- Odoo connection test/save/list with masked responses.
- Odoo model/field browsing and selected-field fetch into DuckDB.
- Pipeline creation from saved script.
- Pipeline enable/disable.
- Manual pipeline run and persisted run detail.

The current build has deliberate gaps that this L8.1 model keeps visible:

- PipelineSchedule fields and scheduler restoration are not implemented yet.
- OdooSyncCursor for incremental/delta sync is not implemented yet.
- Connector pre-step execution in pipelines is not implemented yet.
- FailureNotification ack state and Home failure summary are not implemented yet.
- AgentConversation and AgentGeneratedCode are not implemented yet.
- F-DS-004 table management and F-DS-005 export are not implemented yet.

## Open Questions

No blocking open questions for freezing the v1 data model. The defaults above are sufficient for the bridge-tool scope. Two future-proofing topics are intentionally parked:

- Secret encryption can wait while the app remains a trusted local internal tool.
