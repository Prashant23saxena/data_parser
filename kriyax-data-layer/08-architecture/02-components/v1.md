# Components — KriyaX Data Layer

**Version:** v1
**Status:** FROZEN
**Last updated:** 2026-05-03
**Refs:** L8.1 data model v1, L7 build plan v1
**Implementation baseline checked:** `kriyax-data-app` Phase 1, Phase 2, and initial Phase 3 build slices

## Overview

KriyaX Data Layer should remain a modular monolith: one React/Vite frontend, one FastAPI backend, one local workspace, and one DuckDB warehouse. Components are module boundaries, not deployable microservices.

This matches the existing build:

- FastAPI already uses route modules under `backend/app/api/*`.
- Backend domain logic already lives under `backend/app/services/*`.
- React already exposes all 14 approved screens through one SPA.
- DuckDB, metadata JSON files, scripts, uploads, and run records are all project-local.

The component map below preserves this shape and assigns ownership for both built and remaining work.

## Components

### frontend-app-shell

**Purpose:** Render the desktop-first SPA shell, sidebar navigation, route selection, and screen frame.

**Owns entities:** none.

**Implements features/screens:**
- Screen routing for S-01 through S-14.
- Navigation destinations from L5: Home, Connectors, Catalog, Code Workspace, Saved Scripts, Pipelines, Storage Settings.

**Exposes:**
- React layout components.
- Route/screen selection state.

**Depends on:**
- `frontend-api-client` for health/summary data on shell-level screens.

**Boundaries:**
- Does not own feature-specific API calls or domain state.

### frontend-connectors

**Purpose:** User workflows for file import and Odoo ingest.

**Owns entities:** none; displays and edits transient connector forms.

**Implements features/screens:**
- F-DC-001 Upload CSV/Excel file.
- F-DC-002 Auto-detect columns and editable mapping.
- F-DC-003 Preview before import.
- F-DC-004 Configure Odoo connection.
- F-DC-005 Browse Odoo models and fields.
- F-DC-006 Fetch Odoo records to table.
- F-DC-007 Incremental/delta sync UI entry points.
- F-DC-008 Manage saved connections.
- S-02, S-03, S-04.

**Exposes:**

- Odoo connection/model/field/fetch workflow views.

**Depends on:**
- `backend-imports-api` for file upload/commit.

## Dependency Rules

- Frontend components never read workspace files or DuckDB directly.
- Frontend components call backend APIs through `frontend-api-client`.
- Backend API routers delegate business logic to backend services.
- Backend services may share `backend-workspace-service`.
- Backend services must not mutate another service's owned JSON file except through the owning service.
- `backend-pipelines-service` delegates transformation execution to `backend-execution-service`.
- `backend-pipelines-service` delegates connector pre-step sync to `backend-odoo-service`.
- `backend-agent-service` reads schema context from `backend-catalog-service`; it does not query DuckDB independently unless the catalog service exposes that operation.
- `shared-contract-types` has no runtime dependency on frontend or backend services.

## Cross-Cutting Decisions

### C-001: Modular monolith, not microservices

Keep all backend modules in one FastAPI app. Components are ownership boundaries and contract boundaries, not deployable services.

### C-002: API routers are thin

Routers own HTTP concerns only: request parsing, response shaping, status codes, and error mapping. Domain behavior belongs in `backend/app/services/*`.

### C-003: Workspace service is the only path authority

All services resolve workspace paths through `backend-workspace-service`. Hardcoded workspace file paths outside that service should be avoided.

### C-004: Catalog service is the metadata authority

Any component that needs table metadata should go through `backend-catalog-service`, including the agent and code workspace.

### C-005: Pipelines orchestrate but do not execute internals

Pipelines own scheduling and run records. Execution and connector sync stay delegated to their owning services.

## Current Implementation Alignment

Already aligned:

- `backend/app/api/*` and `backend/app/services/*` already match the router/service split.
- `workspace.py`, `storage.py`, `file_import.py`, `odoo.py`, `execution.py`, and `pipelines.py` already map cleanly to L8.2 backend services.
- `frontend/src/api.ts` already serves as the frontend API client/type boundary.
- `frontend/src/App.tsx` currently contains all screen rendering; future cleanup can split this by frontend component group without changing contracts.

Pending components or component depth:

- `backend-agent-service` and `frontend-agent-panel` are not yet implemented.
- `backend-storage-service` needs table management/export depth.
- `backend-pipelines-service` needs schedule restoration, failure notification state, and Odoo pre-step integration.
- `backend-odoo-service` needs incremental sync cursor support and connection edit/delete.
- Frontend feature modules may stay in `App.tsx` for early speed, but ownership boundaries should be followed when code grows.

## Open Questions

No blocking open questions for L8.2. The component boundaries fit the existing app and the frozen L8.1 data model. A future refactor can split `frontend/src/App.tsx` once the remaining slices are functionally complete.
