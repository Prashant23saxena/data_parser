# UAT Scenarios — KriyaX Data Layer

**Version:** v1
**Status:** FROZEN
**Last updated:** 2026-05-03

## Overview

Seven UAT scenarios verify the L1 promise from the user's view: ingest client data, understand schemas, transform with Python/pandas, schedule refreshes, recover from failures, and export/manage local tables.

## Vision Recap

KriyaX Data Layer is an internal, self-hosted bridge tool for Kriya data engineers and analysts. It ingests CSV/Excel and Odoo data, loads it into queryable tables, helps users write Python/pandas transformations, and schedules recurring updates without Databricks cost.

## Critical Flow Scenarios

### UAT-001: Import CSV and Inspect It in Catalog

**Persona:** Kriya analyst onboarding a client file.
**Goal:** Upload a CSV, confirm schema, create a table, and inspect rows.

Steps:
1. Open the app.
2. Go to Connectors.
3. Choose file import.
4. Upload a CSV with headers and two rows.
5. Review detected columns and types.
6. Rename any duplicate target columns if prompted.
7. Choose target schema `raw_files` and a unique table name.
8. Import.
9. Open Catalog and open the new table.
10. Verify columns and sample rows are visible.

Pass criteria:
- [ ] Import completes without hidden side effects before confirmation.
- [ ] New table appears in Catalog.
- [ ] Table Detail preview shows the expected rows.

### UAT-002: Fetch Odoo Records Into a Local Table

**Persona:** Kriya data engineer connecting client ERP data.
**Goal:** Test an Odoo connection, browse fields, and fetch records.

Steps:
1. Go to Odoo Import Workspace.
2. Enter connection details.
3. Test connection.
4. Search for a model such as `sale.order`.
5. Open field list and select supported fields.
6. Fetch records to a new table.
7. Open the resulting table in Catalog.

Pass criteria:
- [ ] Connection success/failure is clear.
- [ ] Unsupported fields are visibly skipped or warned.
- [ ] Fetched table appears in Catalog with Odoo source.

### UAT-003: Transform Data With Python and Save a Derived Table

**Persona:** Kriya analyst cleaning data.
**Goal:** Load a table, run pandas code, and save a refined table.

Steps:
1. Open Code Workspace.
2. Confirm the imported table appears in available tables.
3. Write Python code using `load_table()`.
4. Create a simple derived DataFrame.
5. Save it with `save_table()`.
6. Run the script.
7. Open Catalog and confirm the derived table exists.

Pass criteria:
- [ ] Script output shows success.
- [ ] Derived table appears in Catalog.
- [ ] Saved script can be reopened and rerun.

### UAT-004: Create, Schedule, and Monitor a Pipeline

**Persona:** Kriya engineer automating a recurring data task.
**Goal:** Create a pipeline from a saved script and schedule it.

Steps:
1. Open Pipelines.
2. Create a pipeline from a saved script.
3. Set daily schedule at 6am in local timezone.
4. Verify next run is shown.
5. Disable and re-enable the pipeline.
6. Manually run it.
7. Open run detail.

Pass criteria:
- [ ] Schedule saves and shows next run.
- [ ] Disabled pipeline is visually distinct.
- [ ] Manual run works even while schedule was disabled.
- [ ] Run detail shows logs, status, duration, and saved tables.

### UAT-005: Recover From a Pipeline Failure

**Persona:** Kriya operator checking daily health.
**Goal:** Notice, inspect, acknowledge, and recover from a failed pipeline.

Steps:
1. Trigger or wait for a pipeline that fails.
2. Open Home.
3. Verify failure summary is visible.
4. Open Pipelines and view failure badge.
5. Open run detail and inspect traceback.
6. Acknowledge the alert.
7. Fix script or run a working script and rerun pipeline.

Pass criteria:
- [ ] Failure is visible without hunting through logs.
- [ ] Error detail is readable.
- [ ] Acknowledgement hides active alert but history remains.
- [ ] New failure reopens alert.

### UAT-006: Run Odoo Sync Before Pipeline Script

**Persona:** Kriya engineer keeping ERP-derived datasets fresh.
**Goal:** Configure a pipeline that syncs Odoo changes before running transformation code.

Steps:
1. Configure Odoo incremental sync.
2. Attach the sync as pipeline pre-step.
3. Run the pipeline.
4. Open run detail.

Pass criteria:
- [ ] Sync step is shown before script step.
- [ ] Sync errors block script execution and show failure.
- [ ] Successful sync then runs script.

### UAT-007: Use Agent to Generate and Correct Python Code

**Persona:** Analyst who knows the data goal but not exact pandas code.
**Goal:** Ask agent for code, insert into editor, and correct an error.

Steps:
1. Open Code Workspace.
2. Open Agent panel.
3. Ask for a transformation using a real catalog table.
4. Insert generated code into editor.
5. Run it.
6. If it fails, click correction action and insert corrected code.

Pass criteria:
- [ ] Agent references real table and column names.
- [ ] Generated code uses `load_table()` and `save_table()`.
- [ ] Code is not auto-run.
- [ ] Correction is proposed after user action.

## Edge / Non-Functional Scenarios

### UAT-010: Empty Workspace Is Understandable

First app launch with no catalog tables should show clear empty states and links to import/connect data.

### UAT-011: Large-ish Local Data Does Not Break the UI

Catalog and preview remain usable with at least 100 tables and a wide table with 100 columns.

### UAT-012: Local-Only Privacy Defaults

Agent context defaults to schema metadata only; sample rows are not sent unless explicitly enabled.
