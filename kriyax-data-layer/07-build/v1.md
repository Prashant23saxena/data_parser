# Build Plan — KriyaX Data Layer

**Version:** v1
**Status:** FROZEN
**Last updated:** 2026-05-02

## Context

- **Build mode:** Local-first internal bridge tool for the Kriya team.
- **Assumed team:** Solo or very small team.
- **Time posture:** Build a working vertical slice first, then deepen reliability.
- **Hard direction:** Python/pandas only for user transformations; no Jupyter, no SQL-first workbench, no heavy orchestrator.
- **Visual direction:** Use L6 v3 images as design-language references only. Use L6 v2 wireframes and L4 specs as the implementation contract.

## Research anchors

- FastAPI remains a strong fit for Python APIs because it is built around Python type hints and automatic API documentation.
- DuckDB remains a strong local analytical database choice because its Python client works directly with Python/DataFrame workflows.
- APScheduler is enough for the first scheduling layer because the product needs interval/cron-style local jobs, not a full Airflow/Dagster DAG platform.
- Vite + React is a good fit for a fast desktop-first internal UI.
- Odoo external APIs remain the main connector integration risk because model metadata and record fetching need careful handling.

## Tech stack

| Layer | Choice | Why |
|---|---|---|
| Frontend | React + TypeScript + Vite | Fast local iteration and enough structure for a dense internal app. |
| UI styling | CSS modules or simple app CSS first; component primitives later if needed | Keeps the first build bare-bones and avoids design-system overhead before flows work. |
| Backend API | FastAPI | Python-native, simple request models, good auto-docs, easy fit with pandas/DuckDB. |
| Data processing | pandas | Matches the user's Python-only transformation direction and the agent output contract. |
| Local execution | Local Python execution wrapper | Saves real `.py` files and executes them with the laptop/project Python while capturing logs/errors and injecting helpers. |
| Storage | DuckDB file in workspace | Lightweight local analytical database for imported and derived tables. |
| File parsing | pandas `read_csv` / `read_excel` | Directly supports CSV/Excel ingestion for the first connector. |
| Odoo connector | Odoo external API client module | Needed for models, fields, records, and delta sync; isolate as a backend service. |
| Scheduling | APScheduler | Good enough for interval/cron local scheduled runs without a full orchestration platform. |
| Metadata | JSON files or small SQLite metadata DB in workspace | Keeps catalog, scripts, connections, and pipeline metadata simple in v1. |
| LLM integration | Provider adapter, initially one provider | Keeps agent work isolated; L8 can define exact contracts. |
| Testing | pytest for backend, Playwright/smoke tests for UI flows | Covers local data flows and screen-level regressions. |

## Build phases

### Phase 0: Foundation and workspace shell — 2-3 days

**Goal:** Create the app skeleton and local workspace contract.

**Build:**
- React/Vite app shell with left sidebar.
- FastAPI backend.
- Workspace folder bootstrap.
- DuckDB health check.
- Basic API client from frontend to backend.
- Screen placeholders for S-01 to S-14 using the approved navigation.

**Screens:** S-01, S-14, app shell.

**Definition of done:**
- App opens locally.
- Backend is always up when app is running.
- Storage Settings can show DuckDB ready/unreachable.
- Sidebar navigation works.

**Screens:** S-02, S-03, S-05, S-06, S-09, S-11 light entry.

### Phase 3: Pipeline scheduling — 4-6 days

**Goal:** Turn saved scripts and Odoo syncs into repeatable local jobs.

**Critical flow covered:** S-11 -> S-12 -> S-13.

**Features:**

### Phase 4: Agentic layer — 5-8 days plus spike

**Goal:** Add schema-aware code generation and self-correction without letting agent complexity block the core data tool.

**Critical flow covered:** S-09 -> S-10 -> S-09, and S-13 -> S-10 -> S-09.

**Features:**
- F-AL-001 Chat interface
- F-AL-002 Schema-aware context loading
- F-AL-003 Generate Python/pandas code
- F-AL-004 Insert code into editor
- F-AL-005 Self-correction on error
- F-AL-006 Conversational follow-ups

**Screens:** S-09, S-10, S-13.

**Definition of done:**
- Agent can read catalog metadata for selected tables.
- Agent generates Python/pandas code only.
- User can insert generated code into editor.
- On traceback, agent can propose a fix using script + error + schema context.
- Self-correction attempts are bounded and visible.

### Phase 5: Robustness and export/management — 4-6 days

**Goal:** Make the tool safe enough for repeated internal use.

**Features:**
- F-DS-004 Basic table management
- F-DS-005 Export table data
- Deepen validation/error/empty states across file import, Odoo, catalog, code, pipelines.

**Screens:** S-07, S-08, plus error/empty states across S-01 to S-14.

**Definition of done:**
- Table rename/truncate/drop are guarded.
- Export CSV works from Catalog/Table Detail.
- Major error states from L4 specs are implemented.
- No flow claims success on partial failure.

## Thin slice explicit exclusions

Not in Phase 1:

- Odoo connection.
- Agentic code generation.
- Scheduled pipelines.
- Incremental/delta sync.
- Destructive table management.
- CSV export.
- Polished empty/error states beyond the thin-slice path.

## Risk register

| Risk | Area | Why it matters | Mitigation |
|---|---|---|---|
| Local Python execution security | Code Workspace | User scripts run on the laptop and can access local files if unrestricted. | Treat as trusted internal tool in v1; document clearly; add workspace-root conventions; consider sandbox later. |
| Helper injection clarity | Code Workspace | Scripts need `load_table()` and `save_table()` without hiding too much. | Generate a small wrapper/context module and keep scripts as normal `.py` files. |
| DuckDB schema model | Storage | Physical schemas vs logical zones affects import/catalog behavior. | Decide in L8.1 data model; default to physical DuckDB schemas if feasible. |
| Excel parsing variability | File import | Sheets, merged cells, date formats, blank headers can break imports. | Start with normal `.csv/.xlsx`; surface mapping/type review; add robust test files. |
| Odoo auth/API variation | Odoo connector | Odoo deployments differ by version/auth/API behavior. | Build isolated Odoo client spike before full S-04; test `fields_get` and `search_read`. |
| Delta sync correctness | Odoo + Pipelines | Incorrect cursor/upsert logic can duplicate or miss records. | Implement after full fetch works; require unique key + modified field; test with controlled fixture. |
| Agent hallucinated code | Agentic Layer | Agent may generate invalid columns or unsafe code. | Keep user-in-the-loop; schema context; insert only, not auto-run; bounded self-correction. |
| Scheduler lifecycle | Pipelines | Jobs must survive app restarts or be restored safely. | Persist pipeline definitions; reload schedules at backend startup; log every run. |
| Generated images text accuracy | Frontend UI | Image text may be approximate. | Use specs/wireframes for exact labels; images only guide density/style. |

## Pillar build order

1. Data Storage foundation.
2. File Data Connectors.
3. Schema & Catalog.
4. Code Workspace.
5. Odoo Data Connectors.
6. Pipeline & Scheduling.
7. Agentic Layer.
8. Table management/export hardening.

## Definition of v1 shipped

V1 is shipped when:

- File import creates new DuckDB tables through target schema + column mapping + preview.
- Odoo connection can fetch selected model/fields into a new table.
- Catalog lists imported and derived tables with metadata and preview rows.
- Code Workspace saves real `.py` files and runs them through local Python execution wrapper.
- Derived tables can be saved and loaded again.
- Pipelines can run saved scripts manually and on schedule with visible history/errors.
- Agent can generate Python/pandas code from schema-aware context and propose fixes for tracebacks.
- Storage Settings shows local DB health.

## Explicitly not in v1

- Client-facing access.
- RBAC/governance/audit platform.
- Full Airflow/Dagster orchestration.
- Full Jupyter/JupyterHub.
- SQL authoring surface.
- Dashboards/charts.
- Connectors beyond CSV/Excel and Odoo.
- Databricks-level scale or production hardening.

## L8 handoff notes

L8 should start with these questions:

1. Are `raw_files`, `raw_odoo`, `staging`, and `curated` physical DuckDB schemas or logical app zones?
2. Should metadata live in JSON files first, or a small SQLite DB from day one?
3. What exact contract does `load_table()` / `save_table()` expose to user scripts?
4. How are Odoo credentials stored locally?
5. What is the agent prompt/context contract for safe Python generation?
