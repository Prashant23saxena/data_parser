# Vision — KriyaX Data Layer

**Version:** v1
**Status:** FROZEN
**Last updated:** 2026-05-01

## The Problem

The Kriya team works with client data that lives in scattered places — CSV/Excel files exported manually, and Odoo ERP systems. There's no unified tool to pull this data together, clean it, define schemas, join across sources, and schedule recurring updates. Every client engagement starts with hours of manual data wrangling in spreadsheets and one-off scripts. Writing SQL across unfamiliar client schemas is slow and error-prone — the team has to manually explore table structures before they can even begin. Databricks solves this perfectly, but it costs too much for the current stage and can't be customized.

## The User

**The Kriya team** — data engineers and analysts who onboard client data, build transformation pipelines, and deliver refined datasets. They operate this tool internally; clients never log in.

**Context of use:** When onboarding a new client's data (connecting to Odoo, uploading CSVs), when building and scheduling ETL pipelines, and when maintaining ongoing data flows for existing clients.

## The Promise

**"An agentic, self-hosted data platform where the Kriya team ingests client data from CSV/Excel and Odoo, loads it into queryable database tables, describes what they need in plain English and an AI agent writes the SQL — joining, cleaning, and transforming data — with scheduled pipeline runs delivering refined datasets. All without Databricks' cost."**

## Success Criteria

- Can onboard a new client's Odoo data in hours, not days of manual scripting
- Scheduled pipelines run daily/hourly without manual intervention
- CSV/Excel files load into proper database tables and are queryable via SQL
- Can write SQL to join across sources (Odoo + CSV) and clean data
- Cleaned/refined tables have visible schemas and can be pulled into other KriyaX layers (e.g., decision engine, reporting) as structured data models
- Schema management is first-class — can view, define, and evolve table schemas as data sources change
- **Agentic-first:** team members describe data needs in plain English; an LLM agent generates the SQL automatically
- The AI agent is schema-aware — reads table metadata, column types, relationships, and available data to write accurate queries
- Entirely self-hosted and open-source — zero licensing cost

## Non-Goals

This app is explicitly **NOT**:

- **Not a real-time streaming platform** — batch and scheduled pipelines are sufficient
- **Not an ML/AI training tool** — purely data plumbing, no model building
- **Not client-facing** — no dashboards, portals, or external access for clients
- **Not a governance platform** — no row-level security, audit logs, or RBAC; the team is small and trust is implicit
- **Not a product to sell** — this is an internal bridge tool for Kriya; when the company scales, it will be replaced by Databricks or equivalent
- **Not a visualization tool** — charts and dashboards are future scope, not in this build

## Tone

**Bare-bones and functional.** Get things done without UI frills. A tool that works reliably, not one that looks pretty. Think "pgAdmin meets Airbyte" — a workhorse, not a showpiece.

## Anchors

**Like:** Databricks SQL workspace — the gold standard for what this should *do* (ingest → schema → SQL → schedule). The UI is aspirational but not required at full fidelity.

**Unlike:** Overly complex orchestrators (full Airflow DAGs), heavyweight notebook environments, or tools that require DevOps expertise to set up.

## Bridge Tool Acknowledgment

This tool has a known shelf life. It's built to solve the immediate pain of data wrangling at Kriya's current scale. When Kriya grows to the point where Databricks (or a similar managed platform) makes financial sense, this tool gets retired gracefully. Therefore: **don't over-engineer. Build for now.**
